## 🚀 Featured Projects

### 💳 Sharded Saga Wallet Service
`Oct 2025` · [GitHub](#)

> *Distributed wallet engineered for atomicity, horizontal scale, and fault tolerance*

**Stack:** `Spring Boot` `Apache ShardingSphere` `MySQL` `Saga Orchestration` `Docker`

| Dimension | Implementation |
|---|---|
| 🔄 Transaction Model | **Saga Orchestration** — 5+ atomic steps with automatic compensation on failure |
| 🗄️ Sharding Strategy | **Apache ShardingSphere** — 2 MySQL shards by `user_id` hash, reducing write contention |
| 🐳 Infrastructure | **3 services** via Docker Compose — reproducible distributed transaction testing |

**Key Engineering Decisions:**
- Saga pattern ensures each transactional step either commits cleanly or triggers a compensating rollback — no partial state leaks
- Horizontal sharding via `user_id` hashing distributes load evenly and eliminates single-shard bottlenecks under concurrent writes
- Fully containerized setup enables consistent local and cloud testing of complex distributed workflows

## 📐 Engineering Philosophy

```
"I think in systems, not just features.
 Before writing the first line of code, I ask:
 What breaks at 10x load? What can't be a single point of failure?
 How does this behave under network partitions?
 Saga or 2PC? Sharding or replication? Reactive or thread-per-request?

 Code is easy. Architecture is hard. I embrace both."
```

**What I care deeply about:**
- ⚙️ Distributed transactions — Saga orchestration, compensating actions, idempotency
- 🗄️ Data scalability — horizontal sharding, replication strategies, read/write separation
- 🔁 Event-driven design — async messaging, Kafka consumer groups, backpressure handling
- 🛡️ Resilience — circuit breakers, retries, graceful degradation under failure
- 📦 Reproducibility — Docker Compose, CI/CD, environment parity between local and cloud
- 🧮 Algorithmic thinking — DSA, competitive programming, performance-first mindset
